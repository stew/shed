# Independent Code Review — 2026-05-17

Fresh-eyes evaluation conducted by an LLM sub-agent with no prior project
context. Acting as a checklist for follow-up work; cross out items as
they're addressed.

## 1. Codebase organization

The `shed-core` / `shed` workspace split is **clean and disciplined**.
`shed-core` has zero I/O, zero TUI deps, and a tight surface area
(`Session`, `Shed`, `Capture`, `FilterSpec`, `Notebook`, `AliasFile`,
`Value`). The core crate is ~2800 lines across 8 files; the binary
handles PTY exec (`exec.rs`), ANSI parsing (`ansi.rs`), and the entire
TUI (`tui.rs`). Module-level rustdoc at the top of every file states
what's in scope and what isn't — that discipline is rare.

The **glaring problem is `crates/shed/src/tui.rs` at 12,035 lines and
416 KB.** It contains 479 functions/methods and 48 type definitions:
`App`, every focus mode's key handler (15 `handle_*_key` variants),
every `draw_*` function (47 of them), the entire filter form state
machine, the tab subsystem, the completion engine (carapace
integration), the readline text-field primitives (`tf_*` family), the
undo/redo plumbing, the OSC52 clipboard + a hand-rolled base64 encoder,
history persistence, and the alias manager. The `App` struct itself
has 50+ fields, most for one-off input bars
(`pin_input`/`pin_cursor`/`pin_input_mode`, `rerun_input`/...,
`write_input`/..., `save_input`/..., `open_input`/..., `cmd_edit_input`/...,
`alias_name_input`/..., `search_input`/..., `rename_tab_input`/...).
Each of these is a `(String, usize, bool)` triple — that pattern alone
justifies an `InputBar` struct and an `enum InputMode { Pin, Rerun,
Write, ... }` to collapse ~30 fields into ~3.

Concrete refactor targets:

- **Extract modules:** `tui/input.rs` (the `tf_*` text field primitives
  + `handle_text_input`, ~400 lines), `tui/completion.rs` (the
  carapace/path/env/alias/slash completion family, ~300 lines around
  tui.rs:3742–4115), `tui/clipboard.rs` (OSC52 + base64, ~90 lines),
  `tui/render.rs` (the 47 `draw_*` functions, ~3000 lines),
  `tui/focus/*.rs` (one per focus). Even moving the ~600 lines of tab
  plumbing out (`TabSlot`, `StashedTab`, `try_handle_tab_key`,
  `begin_rename_tab`) would help.
- **Unify input bars.** The 10+ `*_input_mode: bool` / `*_input: String`
  / `*_cursor: usize` tuples should be a single `Option<InputBar { kind:
  InputKind, text: String, cursor: usize }>`. Today's structure makes
  "where is this input bar handled?" require a global search.

By contrast, the core crate's organization is genuinely good:
`filter.rs` at 1825 lines is the biggest, but it's one cohesive subject
with each filter in its own private `apply_*` function and per-filter
tests beside it.

## 2. Test coverage

**267 passing tests** (198 in `shed`, 69 in `shed-core`, 0 doc-tests).
They run in <100 ms and actually test behavior, not just construction.

What's well-tested:

- **`filter.rs` (50 tests)** — each filter kind has happy-path + at
  least one edge case. `where_drops_rows_with_string_that_doesnt_parse_as_number`,
  `where_bad_regex_still_hard_fails`,
  `from_regex_unnamed_groups_get_underscores`, `where_with_and_or_not`
  show genuine edge-case thinking.
- **`session.rs` (11 tests)** — pin/unpin/eviction/LRU all covered,
  including pinned-skips-eviction and `touch_updates_lru_order`.
- **`exec.rs` (9 tests)** — including truncation respect, alt-screen
  detection, child cancellation via the killer, streaming chunk
  receiver. These exercise real PTY subprocesses.
- **`ansi.rs` (12 tests)** —
  `cr_overwrites_existing_chars_on_same_line`,
  `cargo_style_progress_bar_collapses_to_final_state`,
  `sgr_color_persists_after_overwrite` are exactly the right tests for
  a vte-driven renderer.
- **`tui.rs` (177 tests)** — covers `parse_shed_ref`, history walking +
  persistence to tmpfile, palette filtering, predicate decomposition,
  JSON/CSV/TSV write formats, snapshot creation for `@name`/`%N`,
  tab-bar click hit testing.

What's **untested or thin**:

- **`draw_*` functions** — no rendering tests. Possibly hard, but
  `ratatui::TestBackend` exists and the project uses it nowhere. The
  largest single chunk of code has the least testing.
- **Mouse interaction** — `handle_mouse`, `handle_shift_left_click`,
  `open_body_context_menu` rely on `body_regions` / `click_regions`
  populated during draw. Tests like
  `render_table_records_cell_layouts_per_row_per_column` exist but no
  `handle_mouse` test simulates a click + asserts the resulting state.
- **Focus transitions** — no test that says "in `ShedCursor` focus,
  pressing `e` enters `EditShed`". The 11 different `handle_*_key`
  functions are essentially untested as integrated state machines.
- **Tab system** — the tab stashing/restoring (`TabSlot::stash`/`restore`)
  is critical to correctness and has, AFAICT, no tests. Bugs like
  "switching tabs while a modal is open" would only show up at runtime.
- **Notebook edge cases** — version check exists for the top-level
  version but mid-format breakage isn't exercised.

`filter.rs` is ~50% test code, `tui.rs` is ~26%, and the tested portion
is overwhelmingly the *pure functions* — none of the actually-rendered
or actually-input-handled code paths.

## 3. Documentation

The README is **exceptionally good for a pre-alpha** — 720 lines,
covers concepts, focus model, every keybinding for every mode,
copy/paste with terminal-specific notes, fullscreen handover detection
paths. The concepts section reads like a design doc you could hand to
a contributor.

Rustdoc on the public surface of `shed-core` is also strong — every
public type and pub fn in `lib.rs`, `value.rs`, `shed.rs`, `capture.rs`,
`session.rs`, `filter.rs`, `notebook.rs`, `aliases.rs` has a doc
comment, often with rationale. The filter enum variants each document
edge-case behavior. **This is the best part of the project.**

Gaps:

- **No CONTRIBUTING.md, no CHANGELOG.md, no AGENTS.md/CLAUDE.md.** No
  onboarding doc says "how to add a filter" or "how the focus state
  machine works".
- **`tui.rs` has a great module header but no rustdoc on the `App`
  struct's 50 fields** as a unit — for a second contributor it'd be a
  wall.
- **No doc-tests in `shed-core`** ("0 passed; 0 failed; 0 ignored").
  Filter apply functions would benefit from one-line examples that
  compile.
- **No architecture doc** beyond the README's `Architecture` section.
  A diagram of focus transitions, the event-loop tick, or the data flow
  `Capture → strip_ansi → parser → row filters → renderer` would help.

## 4. NIH vs. dependency hygiene

**Mostly good, with one clear NIH and one clear duplication.**

Reuse done well: `portable-pty`, `vte`, `ratatui`+`crossterm`, `shlex`,
`csv`, `serde_json`, `regex`, `indexmap`, `bytes`, `thiserror`, `tokio`
are all the right choices. `shlex` is used for argv tokenization (not
hand-rolled). Carapace shell-out (tui.rs:3874) for argv1+ completion
leverages a tool that ships 1000+ command specs rather than reinventing.

**Reinvented:**

- **Hand-rolled base64** at tui.rs:6748 (40 lines) for OSC 52 clipboard.
  The comment claims "cheaper than pulling in a dep", but the `base64`
  crate adds ~0 in compile time and is universally idiomatic. Replace
  with `base64::engine::general_purpose::STANDARD.encode(text)` — three
  lines plus an import, removes a custom code path with no isolated
  test coverage.
- **Hand-rolled readline-style editing.** The `tf_*` family at
  tui.rs:3444–3550 reimplements line editing. However, given input
  rendering is bespoke (inline cursor inverted as part of a ratatui
  span) and the editor wants to drive multiple input bars from a single
  state shape, this is more defensible than the base64 case. Still
  worth weighing against `tui-input` or `tui-textarea`.

**Duplication that probably *should* be unified:**

- `crates/shed-core/src/filter.rs:251` defines `strip_ansi` — a 100-line
  char-by-char ANSI stripper with `\r`+`\x1b[K` handling, used by
  parsers to see "final state" of each line.
- `crates/shed/src/ansi.rs:44` defines `parse_to_lines` — a `vte`-driven
  ANSI parser doing essentially the same per-line CSI/CR/erase handling
  but producing styled ratatui `Line`s.

These solve overlapping problems with completely independent code,
including a duplicated `erase_in_line` helper. The `vte` parser in
`ansi.rs` is the right design; `strip_ansi` in core could be re-expressed
by feeding bytes through `vte::Parser` with a stripped `Perform` impl
and would shed ~100 lines while gaining the robustness of a proper
state machine. Real risk that bugs fixed in one diverge from the other.

## 5. What's missing for users/contributors

This is where the project shows its single-developer pre-alpha status
most clearly.

**Missing:**

- **No CI config.** No `.github/workflows/`, no GitLab CI, no anything.
  No automated check that PRs build, that clippy is clean, or that
  tests pass. For a 267-test codebase, this is the lowest-effort
  biggest-win improvement.
- **No `rustfmt.toml`, no `clippy.toml`.** Format/lint policy is
  unstated. `cargo clippy` currently produces **8 warnings and 1
  deny-level error** in test code (the test at tui.rs:9691 uses `3.14`
  which trips `clippy::approx_constant`). The error breaks `cargo
  clippy --all-targets`.
- **No `rust-toolchain.toml`.** Edition 2024 + resolver 3 require a
  recent toolchain; nothing pins that.
- **No CONTRIBUTING.md.** No guidance on filter design, test
  conventions, how to add a builtin.
- **No CHANGELOG.md.** Hard to track what landed when from 80 commits
  alone.
- **No release build / install instructions.** README says "build from
  source" but doesn't say `cargo install --path crates/shed`, doesn't
  show a release-mode invocation, and there's no
  homebrew/cargo-binstall/AUR/etc. footnote.
- **No screenshots or demo gif.** The ASCII demo is good but a single
  GIF would communicate the killer-feature value in 5 seconds.
- **No issue templates, no PR template.**
- **No `examples/` directory or sample notebooks.** A `demo.json`
  shipped in the repo would let a new user `cargo run -- examples/demo.json`
  and immediately see structure.
- **No platform support matrix.** README says "Linux/macOS"; what about
  WSL? Windows native (portable-pty supports it)?
- **No man page / `--help` beyond `usage: shed [NOTEBOOK.json]`.**

**Present and good:** LICENSE (MIT), `.gitignore`, README structure.

## 6. Anything else that jumps out

**Code quality positives:**

- **Zero non-test `.unwrap()` calls in production code paths.** Every
  `.unwrap()` is inside `#[cfg(test)]`. Exceptional discipline for
  single-developer pre-alpha.
- **`expect()` calls outside tests** (15 of them) are nearly all on
  invariants ("non-empty prefix", "inactive tab missing its stash", "at
  least one predicate"). Documented intentions, not laziness.
- **Error types** (`FilterError`, `NotebookError`, `AliasError`,
  `ExecError`) use `thiserror` and distinguish schema vs. data errors
  meaningfully.
- **Undo/redo via `Session::clone()`** leverages `bytes::Bytes`
  refcounting. Structural snapshots are cheap because captures share
  underlying buffers.

**Code smells / concerns:**

- **`String::from_utf8_lossy`** in `strip_ansi` (filter.rs:252)
  materialises a full `Vec<char>` of the entire byte buffer, then
  iterates it. For a 16 MB capture this allocates ~64 MB temporarily.
  The vte-based replacement would stream.
- **Per-frame full pipeline reapplication.** With a 16 MB capture and
  a `from-csv | sort-by` pipeline this could chew CPU. No caching of
  the parsed structured value between frames when the pipeline hasn't
  changed.
- **`spawn_blocking` per command + 8192-byte chunk reads.** Each
  command holds a blocking thread. For users running 20+ commands in
  tabs this saturates the default blocking-thread pool (512).
- **`needs_fullscreen` blacklist** (tui.rs:272) is hardcoded. No way to
  extend without recompiling.
- **Clippy errors in tests.** `cargo clippy --all-targets` doesn't
  pass. Fix the `3.14` and clean up the 6 warnings (useless `format!`,
  `let _ = stmt;`, `map_or(false, ...)`, identical if-blocks,
  `Option.and_then(|x| Some(y))` → `map`).
- **`As-cast f64::INFINITY` and `i32` precision casts** scattered
  across `filter.rs`'s compare code aren't audited for overflow.
- **`App::drop` only kills the active tab + iterates stashed tabs.**
  Comment "the blocking PTY-reader tasks will then exit on their own as
  the slave closes" assumes Unix semantics; on Windows worth verifying.

## Top 3 actionable improvements I'd recommend right now

1. **Add CI** (`.github/workflows/ci.yml`): `cargo build`, `cargo test`,
   `cargo clippy --all-targets --workspace -- -D warnings`, `cargo fmt
   --check`. Fix the existing clippy error at tui.rs:9691 (`3.14` → a
   different float that isn't approximate-PI) and the 8 warnings as
   part of the same PR. Two hours of work; prevents the next 100
   commits from drifting.

2. **Decompose `tui.rs`.** Even a coarse split — `tui/mod.rs` keeping
   `App` and `run`, plus extracted `tui/input.rs` (text-field
   primitives), `tui/completion.rs`, `tui/clipboard.rs` (replace
   hand-rolled base64 with the `base64` crate while you're at it),
   `tui/tabs.rs`, and `tui/render/*.rs` (the 47 `draw_*` functions) —
   would take the largest file from 12K to under 4K. Bonus: collapse
   the 10+ `*_input_mode/text/cursor` field triples into a single
   `Option<InputBar>` enum-tagged by purpose.

3. **Unify the two ANSI parsers.** `shed-core::filter::strip_ansi` and
   `shed::ansi::parse_to_lines` solve overlapping problems with
   independent code. Refactor `strip_ansi` to use `vte::Parser` with a
   stripped `Perform` impl that drops styles but preserves cursor-effect
   logic. Same source of truth, less drift risk, less code. While
   there, fix the `Vec<char>` materialization for large captures.

Honorable mention: add doc-tests to the public `Filter::apply` /
`apply_with_notes` so `shed-core` has at least one runnable example,
and ship `examples/demo.json` so `cargo run -- examples/demo.json`
shows the value prop immediately.
